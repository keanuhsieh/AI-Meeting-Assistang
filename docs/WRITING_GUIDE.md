# Agent Factory 技術文件撰寫指南 (SOP)

---
- **文件目的:** 為 Agent Factory 系列專案提供一套標準化的技術文件撰寫、管理與複製流程。
- **適用對象:** 所有參與專案的開發者、設計師、產品經理。
---

## 1. 理念與目標 (Philosophy & Goals)

1.  **單一事實來源 (Single Source of Truth):** 技術文件是專案設計與實作的最終依據，用以解決團隊成員間的認知差異。
2.  **加速新成員上手 (Accelerate Onboarding):** 新進開發者應能透過閱讀文件，快速理解專案的架構、設計模式與開發規範。
3.  **確保一致性 (Ensure Consistency):** 為衍生專案 (B, C, D...) 提供一個可複用的基礎模板，確保品牌視覺與程式碼風格的統一。
4.  **促進可擴展性 (Facilitate Scalability):** 將「可變」與「不變」的設計決策文檔化，使未來的功能擴充或修改有跡可循。

## 2. 文件結構與位置 (Structure & Location)

所有技術文件都應存放在專案的根目錄下一個名為 `docs` 的資料夾中。

```
/
├── docs/
│   ├── SPECIFICATION.md      // 核心技術與規格文件
│   ├── WRITING_GUIDE.md      // (本文件) 技術文件撰寫指南
│   ├── CHANGELOG.md          // 版本變更日誌
│   └── assets/               // 存放文件中引用的圖片、圖表
├── public/
├── src/
└── ...
```

- **`SPECIFICATION.md`:** 核心規格書，記錄從設計理念到元件實作的所有細節。
- **`WRITING_GUIDE.md`:** (本文件) 關於如何撰寫與維護 `docs` 資料夾內所有文件的 SOP。
- **`CHANGELOG.md`:** 記錄每次對專案 **有感** 變更的日誌，應遵循 [Keep a Changelog](https://keepachangelog.com/zh-TW/1.0.0/) 格式。
- **`docs/assets/`:** 所有在文件中使用的截圖、架構圖等，都應存放在此。

## 4. 從 A 專案到 B 專案的工作流程 (A -> B Workflow)

此流程定義如何利用現有文件，快速啟動一個新的衍生專案 (Project B)。

1.  **複製 (Fork):**
    *   開發者將 A 專案的 `docs` 資料夾 **完整複製** 到 B 專案的根目錄。

2.  **研讀 (Study):**
    *   開發者詳細閱讀 B 專案中的 `docs/SPECIFICATION.md` (此刻內容仍是 A 專案的)。**目標是完全理解 A 專案的設計理念與技術實作**，將其內化為基礎知識。

3.  **需求訪談 & 差異分析 (Requirement & Delta Analysis):**
    *   開發者與產品經理/客戶開會，**以 A 專案的規格文件為討論基礎**，逐項詢問 B 專案的需求與差異點。
    *   **範例問題:**
        *   "A 專案的主色是橘色，B 專案的主色是什麼？"
        *   "B 專案需要『客戶見證』這個區塊嗎？"
        *   "B 專案有 A 專案沒有的新區塊嗎？"

4.  **文件修改 (Update Documentation):**
    *   開發者根據差異分析的結果，**先修改 B 專案的 `SPECIFICATION.md`**，而不是直接寫程式。
    *   **執行動作檢查清單 (Action Items Checklist):**
        *   [ ] 更新文件開頭的元數據：`專案名稱`、`版本`、`基於專案` 等。
        *   [ ] 更新 `設計系統` 章節中的設計令牌 (顏色、字體等)。
        *   [ ] 新增、刪除或修改 `元件庫` 章節。
        *   [ ] **【重要】** 更新 `部署與營運` 章節中的 **分析追蹤 ID** (GTM, Meta Pixel)。
        *   [ ] **【重要】** 更新 `部署與營運` 章節中的 **外部連結** (如購買連結)。

5.  **依據文件開發 (Develop from Spec):**
    *   現在，B 專案的 `SPECIFICATION.md` 已經成為了該專案的「單一事實來源」。
    *   開發者**依照這份新文件**開始進行開發，確保每一行程式碼都符合事先定義好的規範。

> **特別注意：專案特定規範 (Project-Specific Mandates)**
> 
> 請留意 `SPECIFICATION.md` 中可能包含針對特定專案系列的強制性規範（例如，要求特定元件的程式碼必須完全複製，或採取特定的資源管理方式）。這些規範是為了確保核心功能（如品牌信任、金流安全）在所有專案中的絕對一致性，其優先級高於通用的最佳實踐。
